# Telegram Bot Integration Architecture

## Executive Summary

Analysis of the Telegram gateway integration in the self-hosted Hermes Agent platform, covering the adapter architecture, message lifecycle, session management, and delivery paths. The system uses a two-layer architecture: platform adapters handle protocol-specific I/O, while the gateway routes messages through a per-chat session store to the AIAgent for processing.

## Main Content

### Architecture Overview

The Telegram gateway runs as a single background process (systemd user service) that maintains a persistent connection to Telegram's Bot API. Messages flow through three stages:

1. **Adapter Layer** (`plugins/platforms/telegram/adapter.py`) — Handles Bot API communication, message formatting, media attachments, and platform-specific features like topics/threads, typing indicators, and inline keyboards.

2. **Gateway Layer** (`gateway/run.py`) — Routes messages to sessions, manages agent lifecycle, handles interrupts, and coordinates delivery between the stream consumer and tool-progress systems.

3. **Agent Layer** (`run_agent.py`) — The Hermes AIAgent processes messages through the LLM, executes tool calls, and streams responses back through the gateway.

### Message Delivery Paths

There are 10 distinct code paths that can emit Telegram messages per user request:

| Path | Source | Purpose |
|------|--------|---------|
| Tool progress | `send_progress_messages()` | 💻🔍📄 emoji status bubbles |
| Interim commentary | `_send_commentary()` | Mid-turn agent commentary |
| Stream initial message | `_send_or_edit()` first-send | First text chunk of response |
| Overflow split | `_send_new_chunk()` | "(1/2)" continuations >4096 chars |
| Fallback final | `_send_fallback_final()` | Continuation on edit failure |
| Tail flush | `_flush_segment_tail_on_edit_failure()` | Dropped text before tool boundary |
| Long heartbeat | Gateway timer | ⏳ Working — N min notifications |
| Post-agent final | `already_sent` gate | Normal final response delivery |
| Footer line | Post-stream metadata | Runtime footer |
| Queued follow-up | `_run_agent` re-entry | First response before next turn |

### Configuration

Telegram-specific display settings are controlled under `display.platforms.telegram` in `config.yaml`:

```yaml
display:
  platforms:
    telegram:
      streaming: true
      tool_progress: false
      interim_assistant_messages: false
      long_running_notifications: false
      cleanup_progress: true
```

Resolution order: per-platform override → global setting → built-in platform default.
