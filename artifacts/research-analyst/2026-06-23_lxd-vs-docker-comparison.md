# LXD Container vs Docker Comparison

## Executive Summary

Both LXD and Docker provide containerization, but they serve fundamentally different use cases. LXD provides system containers (full OS isolation, runs its own init system) and is better suited for long-running infrastructure workloads. Docker provides application containers (single-process, ephemeral) and is better for microservice deployments. For agent-hosting environments, LXD is preferred because agents need a persistent OS environment with SSH access, systemd, and long-running daemon support.

## Key Findings

1. **LXD advantages:** Full OS isolation, systemd, SSH, persistent storage, live migration, lower overhead for system workloads.
2. **Docker advantages:** Ecosystem maturity, image registry (Docker Hub), orchestration (Kubernetes), ephemeral workload optimization.
3. **Agent workloads:** LXD is better for Hermes agents — need init system, persistent sessions, SSH access.
4. **Image management:** LXD images are larger (Ubuntu ~200MB vs Docker ~80MB) but include a full OS.
5. **Network model:** LXD uses bridge networking with NAT by default; Docker provides overlay networks with DNS service discovery.

## Conclusions

For the agent-env-selfhosted platform, LXD system containers are the correct choice. Docker remains preferred for tool-specific microservices (Redis, PostgreSQL) that run adjacent to the agent containers.

## Uncertainties & Gaps

- LXD clustering at scale (>10 nodes) not yet evaluated
- Docker-in-LXD pattern (running Docker inside LXD containers) not yet benchmarked for performance
